### Эталон
  
  Для построения эталона были использованы траектории полученные Outside-in по маркеру на камере и Inside-out по маркерам на столе.
  Данные двух траекторий были слиты в одну. Сглаживание траектории было выполнено при помощи алгоритма скользящего окна.
  
  Оценка точности эталона производилась при помощи метода [кросс-валидации](https://www.mdpi.com/1424-8220/25/16/4908): случайным образом из Траектории 1 исключались 20% данных, после чего на оставшихся 80% вычислялась усредненная матрица перехода. Далее матрица перехода применялась к исключенной выборке и находилось СКО (RMS) между найденной точкой и точкой из траектории.

  Полученные результаты:

     [Pseudo-GT Валидация]:
    -> СКО исходных траекторий:  14.71 мм | 1.121°
    -> СКО псевдо-эталона с outside_in:       4.89 мм | 0.496°
    -> СКО псевдо-эталона с inside_out:       12.89 мм | 1.121°
    -> Дисбаланс трансляции (смещение к источнику): 8.00 мм
   
  Из оценки погрешностей следует, что псевдо-эталон лег между двумя траекториями, но имеет сильный перекос к траектории outside_in. Объясняется это тем, что точек в данной траектории сильно больше, чем в траектории inside_out. Убедимся в этом наглядно, визуализировав данные траектории.

## Часть 1
### Camera-only трекинг
Восстановление траектории по видеопотоку было произведено с помощью стандартного метода ```o3d.pipelines.odometry.compute_rgbd_odometry()``` библиотеки Open3D. Выравнивание траектории в пространстве было выполнено при помощи алгоритма Umeyama с использованием псевдо-эталонной траектории. Восстановление было произведено на отдельных чанках, чтобы исключить ошибку, вызванную скачками кадров.

 - **Метод Inside-out** показывает лучший результат по всем параметрам. Но на графиках видно, что он достаточно сильно шумит. Основные слабые места данного метода - позиция и расстояние до маркера. 
Чем дальше маркер - тем меньше пискелей на кадре он занимает, а значит и ниже точность. Возникает явление субпиксельного шума. Так же данный метод плохо работает при быстрых движениях и поворотах из-за смаза кадра.

- **Метод Outside-in.** 
 Показал себя менее точным чем метод Inside-out. Подвержен все тем же проблемам, что и Inside-out. Так же из-за того что маркер 
 всего один более вероятен случай планарного вырождения.


- **Метод RGB-D одометрии.** Показал худшие результаты. Это можно связать с тем, что исходные данные лидара достаточно сильно шумели. Так же можно заметить, что данный 
 алгоритм дает большую ошибку на кадрах без движения, связано это с тем, что при отсутствии движения вырождается геометрическая оптимизация
 и из-за этого реальный поворот не считывается.

## Часть 2. 3D-реконструкция сцены
Для реконструкции были использованы два метода: Point cloud fusion (накладывание облака точек друг на друга на каждом кадре) и TSDF Integration (реконструкция с помощью воксельной сетки). Каждый из методов был задействован как с траекторией, так и без. Посмотрим на визуализации 3D-сцен. Был выбран чанк с плавным движением с маркером.
<img width="800" height="307" alt="tsdf_comp (2)" src="https://github.com/user-attachments/assets/eab13828-8ae3-43c5-84f6-9e137a2dc39a" />

Реконструкция с помощью метода TSDF без траектории в качестве prior (слева) и с prior (справа).

<img width="800" height="307" alt="point_cloud_fusion_cmp" src="https://github.com/user-attachments/assets/5f1e55e6-36a6-434a-851c-546c4ac068ca" />
Реконструкция с помощью метода Point Cloud Fusion без траектории в качестве prior (слева) и с prior (справа).

 Для метода TSDF траектория улучшила качество поверхностей, убрало рябь, объекты стали иметь более ясные контуры.   

Для метода Point Cloud Fusion траектория позволила более точно спозиционировать кластеры точек для каждого кадра в глобальной сцене, что сильно
повлияло на качество и узнаваемость сцены, хотя сами поверхности и геометрия далеки от идеалаю.

Из визуализаций видно, что лучший результат достигается методом TSDF с траекторией Inside-out в качестве prior.
